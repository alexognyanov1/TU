## 1. Общ преглед

Системата за дистанционно управление на гаражна врата е embedded IoT проект, който позволява на потребителя да отваря и затваря гаражната си врата дистанционно от всеки браузър. Системата се състои от три основни части:

- **Backend сървър** — Python приложение, което стои в центъра на комуникацията и управлява логиката.
- **ESP32 микроконтролер** — вграден в гаражната врата, управлява физическия мотор.
- **Браузърен интерфейс** — HTML/JavaScript страница, достъпна от всяко устройство с браузър.

Всички компоненти комуникират в реално време чрез WebSocket протокол, а потребителите се аутентикират чрез JWT токени. Системата поддържа множество едновременни браузърни клиенти, като всички получават синхронизирано текущо състояние.

## 2. Архитектура на системата

```
┌──────────────────────────────────────────────────────────────────────┐
│                      Домашна мрежа / Интернет                        │
└──────────────────┬───────────────────────────────┬───────────────────┘
                   │                               │
         ┌─────────▼──────────┐        ┌───────────▼──────────┐
         │   FastAPI Backend  │        │     ESP32 Device     │
         │   (Python сървър)  │◄──────►│  (Гаражна врата)     │
         │   Порт 8000 / 443  │ WebSoc │  (WiFi + Мотор)      │
         └─────────▲──────────┘ -kets  └───────────┬──────────┘
                   │                               │
          HTTP / WebSocket                   GPIO / PWM
         ┌─────────▼──────────┐        ┌───────────▼──────────┐
         │    Браузър (UI)    │        │   DC Мотор           │
         │  - Вход (Login)    │        │   (H-Bridge)         │
         │  - Контролен панел │        └──────────────────────┘
         └────────────────────┘
```

Сървърът играе роля на **централен медиатор** — всяка комуникация (команди от потребителя, статус от устройството) минава през него. Директна връзка между браузъра и ESP32 не съществува.

## 3. Компоненти

### 3.1 Backend (Сървър)

Сървърът е написан на **Python** с помощта на **FastAPI** framework. Изпълнява се чрез **Uvicorn** ASGI сървър.

| Файл | Описание |
|------|----------|
| `main.py` | Входна точка на приложението. Дефинира WebSocket endpoints и сервира HTML файловете. |
| `auth.py` | Модул за аутентикация. Управлява JWT токени и route за вход. |
| `door.py` | Endpoint за задействане на мотора (`POST /door/trigger`). |
| `state.py` | Endpoint за четене на текущото състояние (`GET /door/state`). |
| `ws_manager.py` | Централна клас-базирана логика. Управлява WebSocket връзките и разпраща (broadcasts) съобщения. |
| `config.py` | Конфигурация чрез Pydantic Settings. Чете от `.env` файл. |
| `requirements.txt` | Списък с Python зависимости. |
| `.env` | Конфигурационен файл с credentials и тайни ключове. |

**Клас `ESP32Manager` (`ws_manager.py`)** е сърцето на сървъра. Той:

- Поддържа активната WebSocket връзка до ESP32 устройството.
- Следи текущото състояние на вратата (`door`: open / closed / unknown) и мотора (`motor`: idle / moving).
- Поддържа списък с всички свързани браузърни клиенти.
- Broadcast-ва актуалното състояние до всички браузъри при всяка промяна.
- Изпраща команди до ESP32.

---

### 3.2 Embedded (ESP32 микроконтролер)

Firmware-ът е написан на **C++** за Arduino среда. Файлът е `GarageDoor.ino`.

Основни задачи на ESP32:
- Свързване с WiFi мрежата.
- Поддържане на WebSocket връзка към backend сървъра.
- Получаване на команди от сървъра и управление на физическия DC мотор.
- Изпращане на текущото си състояние към сървъра.
- Запазване на конфигурацията и последното известно състояние на вратата в NVS (Non-Volatile Storage).

---

### 3.3 Frontend (Потребителски интерфейс)

Потребителският интерфейс се сервира директно от FastAPI сървъра като статични HTML файлове.

| Файл | Описание |
|------|----------|
| `login.html` | Страница за вход. Изпраща username и password, получава JWT токен и го записва в `localStorage`. |
| `index.html` | Главен контролен панел. Показва статус на устройството, вратата и мотора. Позволява задействане на вратата. |

**Функционалности на контролния панел:**
- Индикатор за онлайн/офлайн статус на ESP32.
- Badge за състоянието на вратата (Отворена / Затворена / Неизвестно) с цветово кодиране.
- Badge за мотора (Idle / Moving) с анимация при движение.
- Бутон за задействане с динамичен надпис (Отвори / Затвори / Задействай).
- Текстови съобщения за обратна връзка при команди.
- Бутон за изход (Logout).

## 4. Комуникационни протоколи

Системата използва **два комуникационни протокола**:

### 4.1 HTTP / REST API

Използва се за:
- Зареждане на HTML страниците (GET заявки).
- Аутентикация на потребителя (POST `/auth/login`).
- Изпращане на команда за задействане на вратата (POST `/door/trigger`).
- Четене на текущото състояние (GET `/door/state`).

REST заявките от браузъра включват JWT токен в `Authorization` header-а:

```
Authorization: Bearer <jwt_token>
```

### 4.2 WebSocket

WebSocket протоколът осигурява **постоянна двупосочна (full-duplex) връзка** в реално време.

Системата поддържа два отделни WebSocket endpoint-а:

| Endpoint | Участници | Посока | Цел |
|----------|-----------|--------|-----|
| `/ws/esp` | ESP32 ↔ Backend | Двупосочна | ESP32 изпраща статус; Backend изпраща команди |
| `/ws/ui` | Backend → Браузър | Еднопосочна (сървър → клиент) | Backend broadcast-ва статуса до всички браузъри |

**Защо WebSocket?**
- Позволява незабавно push известяване без polling.
- Идеален за real-time синхронизация между множество браузъри.
- Поддържа постоянна връзка с устройството без нужда от многократно handshake.

Локалната среда използва `ws://`, а продукционната среда поддържа `wss://` (WebSocket over TLS) на порт 443.

## 5. Формат на данните

Всички съобщения са в **JSON формат** — и при REST, и при WebSocket комуникацията.

### 5.1 REST съобщения

**Заявка за вход:**
```json
{
  "username": "admin",
  "password": "changeme"
}
```

**Отговор при успешен вход:**
```json
{
  "access_token": "<jwt_string>",
  "token_type": "bearer"
}
```

**Отговор на `GET /door/state`:**
```json
{
  "door": "open",
  "motor": "idle",
  "esp_connected": true
}
```

Полетата на отговора:

| Поле | Тип | Стойности | Описание |
|------|-----|-----------|----------|
| `door` | string | `"open"`, `"closed"`, `"unknown"` | Текущо положение на вратата |
| `motor` | string | `"idle"`, `"moving"` | Дали моторът работи в момента |
| `esp_connected` | boolean | `true`, `false` | Дали ESP32 е свързан към сървъра |

**Отговор на `POST /door/trigger` (успех):**
```json
{
  "status": "triggered"
}
```

---

### 5.2 WebSocket съобщения

**ESP32 → Backend (статус съобщение):**

Изпраща се при всяка промяна в състоянието на вратата или мотора.

```json
{
  "door": "open",
  "motor": "idle"
}
```

И двете полета са опционални — може да се изпрати само едното при частична промяна.

**Backend → ESP32 (команда):**

```json
{
  "action": "trigger"
}
```

В момента системата поддържа само действие `"trigger"`. ESP32 определя посоката на движение въз основа на текущото си вътрешно състояние на вратата.

**Backend → Браузър (broadcast):**

Изпраща се при всяка промяна в системата, както и незабавно след свързване на нов браузърен клиент.

```json
{
  "door": "closed",
  "motor": "idle",
  "esp_connected": true
}
```

---

### 5.3 JWT токен

JWT токенът съдържа следния payload:

```json
{
  "sub": "admin",
  "exp": 1742567890
}
```

| Поле | Описание |
|------|----------|
| `sub` | Username на потребителя |
| `exp` | Unix timestamp на изтичане (24 часа след издаване) |

Алгоритъм: **HS256** (HMAC SHA-256)
Валидност: **24 часа**
Ключ: стойността на `JWT_SECRET_KEY` от `.env` файла

## 6. Потоци от данни (Data Flows)

### 6.1 Вход в системата

```
1. Потребителят отваря /login в браузъра
        ↓
2. Браузър: POST /auth/login
   {"username": "admin", "password": "changeme"}
        ↓
3. Backend: Проверява credentials спрямо config
        ↓
4. Backend: Генерира JWT токен (24h), връща {"access_token": "...", "token_type": "bearer"}
        ↓
5. Браузър: Записва токена в localStorage
        ↓
6. Браузър: Пренасочва към /
        ↓
7. JavaScript: Установява WebSocket връзка към /ws/ui?token=<JWT>
        ↓
8. Backend: Валидира JWT, добавя браузъра в списъка с клиенти
        ↓
9. Backend: Изпраща текущото пълно състояние на системата
        ↓
10. Браузър: Рендира UI с получените данни
```

---

### 6.2 Инициализация на ESP32

```
1. ESP32 се включва
        ↓
2. Зарежда конфигурацията от NVS (WiFi, backend host/port/token, door state)
        ↓
3. Опитва се да се свърже с WiFi (таймаут: 15 секунди)
        ↓
   ┌─── Успех ──────────────────────────────────────────────────────┐
   │                                                                 │
   4a. Проверява интернет свързаност (ping 8.8.8.8 / 1.1.1.1)    │
        ↓                                                           │
   5a. Свързва се към /ws/esp?token=<wsToken>                     │
        ↓                                                           │
   6a. Изпраща начален статус: {"door":"closed","motor":"idle"}   │
        ↓                                                           │
   7a. Влиза в главния loop (слуша за команди, reconnect при нужда)│
   └────────────────────────────────────────────────────────────────┘

   ┌─── Неуспех (таймаут) ──────────────────────────────────────────┐
   │                                                                  │
   4b. Стартира AP: "GarageDoor-Setup" / "setup1234"               │
        ↓                                                            │
   5b. Стартира HTTP уеб сървър (порт 80) с форма за конфигурация  │
        ↓                                                            │
   6b. Потребителят попълва SSID, парола, host, port, token        │
        ↓                                                            │
   7b. ESP32 записва конфигурацията в NVS и рестартира            │
   └────────────────────────────────────────────────────────────────┘
```

---

### 6.3 Задействане на вратата

```
1. Потребителят натиска бутона в браузъра
        ↓
2. Браузър: POST /door/trigger
   Header: Authorization: Bearer <JWT>
        ↓
3. Backend: Валидира JWT (require_auth dependency)
        ↓
4. Backend: Проверява дали ESP32 е свързан
   → Ако не: HTTP 503 Service Unavailable
        ↓
5. Backend: Проверява дали моторът вече работи
   → Ако да: HTTP 409 Conflict
        ↓
6. Backend: Изпраща {"action": "trigger"} към ESP32 по WebSocket
        ↓
7. ESP32: Получава командата, парсва JSON
        ↓
8. ESP32: Определя посоката:
   - door == "closed"  → FORWARD (отваря)
   - door == "open"    → BACKWARD (затваря)
   - door == "unknown" → FORWARD (по подразбиране)
        ↓
9. ESP32: Задава GPIO пинове за посока и включва PWM
        ↓
10. ESP32: Изпраща {"motor": "moving"} → Backend → всички браузъри
        ↓
11. ESP32: Изчаква 2000ms
        ↓
12. ESP32: Спира мотора (пинове LOW, PWM = 0)
        ↓
13. ESP32: Обновява doorState, записва в NVS
        ↓
14. ESP32: Изпраща {"door": "open", "motor": "idle"}
        ↓
15. Backend: Обновява вътрешното си състояние
        ↓
16. Backend: Broadcast {"door":"open","motor":"idle","esp_connected":true}
        ↓
17. Всички браузъри: Обновяват UI в реално време
```

## 7. Аутентикация и сигурност

### 7.1 JWT аутентикация (потребителски интерфейс)

Потребителите на уеб интерфейса се аутентикират чрез **JWT (JSON Web Token)**. Процесът е:

1. Потребителят въвежда username и password.
2. Сървърът валидира credentials и издава подписан JWT.
3. JWT се съхранява в `localStorage` на браузъра.
4. При всяка REST заявка токенът се изпраща в `Authorization: Bearer` header.
5. При свързване по WebSocket токенът се изпраща като query параметър: `/ws/ui?token=<JWT>`.
6. Сървърът валидира токена при всяка заявка чрез `require_auth` dependency.

JWT е с валидност **24 часа** и се подписва с HS256 алгоритъм.

---

### 7.2 Shared Token аутентикация (ESP32)

ESP32 устройството не използва JWT. Вместо това се използва **споделен секретен токен** (`ESP_WS_TOKEN`), конфигуриран едновременно в сървъра (`.env`) и в ESP32 (NVS). При свързване ESP32 изпраща токена като query параметър:

```
ws://host:port/ws/esp?token=<ESP_WS_TOKEN>
```

Ако токенът е невалиден, сървърът затваря WebSocket връзката с код `1008 Policy Violation`.

---

### 7.3 TLS / криптирана връзка

Системата поддържа **TLS (Transport Layer Security)** за криптирана комуникация:

- При продукционна среда се използва порт **443** и протокол `wss://` (WebSocket Secure).
- ESP32 използва `WiFiClientSecure` библиотека за TLS.
- ESP32 не верифицира CA сертификата на сървъра (приемливо за домашна мрежа).

---

### 7.4 Бележки по сигурността

| Аспект | Оценка | Бележка |
|--------|--------|---------|
| JWT аутентикация | Добра | 24-часова валидност, HS256 подпис |
| Shared token за ESP32 | Приемливо | Подходящо за домашна мрежа |
| TLS поддръжка | Добра | `wss://` за продукция |
| Credentials в `.env` | Приемливо | Само за локално/домашно ползване |
| Липса на rate limiting | Слабост | Уязвимост към brute-force при вход |
| CA верификация (ESP32) | Слабост | Потенциална MITM уязвимост |
| Единствен потребител | Ограничение | Не поддържа multiple users |
| Слаб JWT secret в примера | Слабост | Трябва да се смени в продукция |

## 8. Управление на хардуера

### 8.1 GPIO конфигурация

ESP32 управлява DC мотора чрез H-Bridge драйвер, свързан към три GPIO пина:

| Пин | GPIO | Функция |
|-----|------|---------|
| MOTOR_PIN1 | GPIO 27 | Посока 1 (Forward) |
| MOTOR_PIN2 | GPIO 26 | Посока 2 (Backward) |
| ENABLE_PIN | GPIO 14 | PWM сигнал (скорост) |

---

### 8.2 PWM и управление на мотора

Скоростта на мотора се контролира чрез **PWM (Pulse Width Modulation)**:

| Параметър | Стойност |
|-----------|----------|
| PWM честота | 30 000 Hz (30 kHz) |
| PWM резолюция | 8 бита (0–255) |
| Работен Duty Cycle | 200 / 255 (~78.4%) |
| Продължителност на работа | 2000 ms (2 секунди) |

---

### 8.3 Логика на движението

| Текущо состояние | Команда | Посока | Нов статус |
|-----------------|---------|--------|-----------|
| `"closed"` | trigger | FORWARD | `"open"` |
| `"open"` | trigger | BACKWARD | `"closed"` |
| `"unknown"` | trigger | FORWARD | `"unknown"` (не се обновява) |

**Последователност при задействане:**
1. Установяване на `motorBusy = true`
2. Изпращане на `{"motor": "moving"}` към сървъра
3. Задаване на GPIO пиновете за правилна посока
4. Включване на PWM с duty cycle 200
5. Изчакване 2000 ms
6. Деактивиране на PWM и GPIO пинове (мотор спира)
7. Обновяване на `doorState`
8. Запазване на новото състояние в NVS
9. Изпращане на `{"door": "<new_state>", "motor": "idle"}` към сървъра
10. Установяване на `motorBusy = false`

---

## 9. Управление на WiFi и първоначална настройка

### 9.1 Нормален режим

При стартиране ESP32 зарежда записаните в NVS WiFi credentials и се опитва да се свърже. Таймаутът е **15 секунди**. При успех продължава с проверка на интернет и свързване към сървъра.

---

### 9.2 Setup режим (Captive Portal)

Ако WiFi свързването неуспее (липсват credentials или мрежата не е достъпна), ESP32 влиза в **Setup Mode**:

1. Стартира WiFi Access Point:
   - **SSID:** `GarageDoor-Setup`
   - **Парола:** `setup1234`
2. Стартира HTTP сървър на порт **80**.
3. Хоства HTML форма (Captive Portal), в която потребителят въвежда:
   - WiFi SSID
   - WiFi парола
   - Backend host (IP адрес или домейн)
   - Backend port (напр. `8000` или `443`)
   - WebSocket token
4. След изпращане на формата, ESP32 записва данните в NVS и рестартира.

---

### 9.3 Проверка на интернет свързаност

Преди да се опита да се свърже към backend сървъра, ESP32 проверява дали има интернет достъп чрез **ping** до:
- `8.8.8.8` (Google DNS)
- `1.1.1.1` (Cloudflare DNS)

Тази стъпка гарантира, че WiFi е свързан не само до рутера, но и до интернет.

---

## 10. Управление на WebSocket връзката

### 10.1 Свързване

ESP32 изпраща WebSocket handshake заявка към:

```
ws://<wsHost>:<wsPort>/ws/esp?token=<wsToken>
```

или при TLS:

```
wss://<wsHost>:<wsPort>/ws/esp?token=<wsToken>
```

При успешно свързване ESP32 незабавно изпраща текущото си известно състояние (door + motor), за да синхронизира сървъра.

---

### 10.2 Reconnection с exponential backoff

При прекъсване на връзката ESP32 автоматично се опитва да се reconnect с нарастваща пауза:

| Опит | Пауза |
|------|-------|
| 1 | 2 секунди |
| 2 | 4 секунди |
| 3 | 8 секунди |
| ... | ... |
| N | 60 секунди (максимум) |

При успешно свързване паузата се нулира обратно на 2 секунди.

При disconnected ESP32, сървърът автоматично:
- Маркира `esp_connected = false`
- Нулира motor state на `"idle"`
- Broadcast-ва обновеното съобщение до всички браузъри

---

## 11. Управление на състоянието (State Management)

### 11.1 Сървърно състояние

Цялото централно състояние се пази в паметта чрез класа `ESP32Manager`:

| Поле | Тип | Описание |
|------|-----|----------|
| `ws` | WebSocket | Активна WebSocket връзка до ESP32 (или None) |
| `door` | string | Последно известно положение: `"open"` / `"closed"` / `"unknown"` |
| `motor` | string | Текущо Motor state: `"idle"` / `"moving"` |
| `ui_clients` | list | Списък с всички свързани браузърни клиенти |

Сървърното състояние е **ephemeral (временно)** — не се записва никъде и се губи при рестарт на сървъра. Истинското source of truth за физическото положение на вратата е самия ESP32 (чрез NVS).

---

### 11.2 Персистентно състояние на ESP32

ESP32 използва **NVS (Non-Volatile Storage)** — Flash памет, достъпна чрез `Preferences` библиотека. Пространството от имена (namespace) е `"garage"`.

| Ключ | Тип | Описание | Персистентен |
|------|-----|----------|-------------|
| `ssid` | String | WiFi SSID | Да |
| `wifiPass` | String | WiFi парола | Да |
| `wsHost` | String | Backend hostname/IP | Да |
| `wsPort` | Int | Backend port | Да |
| `wsToken` | String | WebSocket auth token | Да |
| `door` | String | Последно известно положение на вратата | Да |

Полето `door` се обновява след всяко успешно движение на мотора, така че при рестарт на устройството последното известно положение се запазва.

---

### 11.3 Браузърно (client-side) състояние

Браузърът пази **JWT токена** в `localStorage`. Всички останали данни (door state, motor state, esp_connected) идват в реално време от WebSocket и не се кешират локално.

При всяко ново свързване на браузър, сървърът незабавно изпраща текущото пълно състояние на системата.

---

## 13. API Reference

### 13.1 HTTP Endpoints

#### `GET /`
Сервира главния контролен панел (`index.html`).
**Аутентикация:** Не е необходима за зареждане на HTML. JWT се проверява от JavaScript при WebSocket свързване.

---

#### `GET /login`
Сервира страницата за вход (`login.html`).
**Аутентикация:** Не е необходима.

---

#### `POST /auth/login`
Аутентикира потребителя и издава JWT токен.

**Заявка:**
```json
{
  "username": "string",
  "password": "string"
}
```

**Отговор (200 OK):**
```json
{
  "access_token": "<jwt_string>",
  "token_type": "bearer"
}
```

**Грешки:** `401 Unauthorized` при невалидни credentials.

---

#### `POST /door/trigger`
Изпраща команда до ESP32 за задействане на мотора.

**Аутентикация:** `Authorization: Bearer <JWT>`

**Отговор (200 OK):**
```json
{
  "status": "triggered"
}
```

**Грешки:**
- `401 Unauthorized` — невалиден/изтекъл JWT
- `503 Service Unavailable` — ESP32 не е свързан
- `409 Conflict` — моторът вече работи

---

#### `GET /door/state`
Връща текущото известно состояние на системата.

**Аутентикация:** `Authorization: Bearer <JWT>`

**Отговор (200 OK):**
```json
{
  "door": "open | closed | unknown",
  "motor": "idle | moving",
  "esp_connected": true | false
}
```

---

### 13.2 WebSocket Endpoints

#### `ws://<host>/ws/esp?token=<ESP_WS_TOKEN>`
WebSocket за ESP32 устройството.

- **Аутентикация:** Query параметър `token` (Shared Token)
- **Посока:** Двупосочна
- **ESP32 → Backend:** `{"door": "...", "motor": "..."}`
- **Backend → ESP32:** `{"action": "trigger"}`
- **При грешен token:** Затваря се с код `1008`

---

#### `ws://<host>/ws/ui?token=<JWT>`
WebSocket за браузърни клиенти.

- **Аутентикация:** Query параметър `token` (JWT)
- **Посока:** Еднопосочна (сървър → браузър)
- **Backend → Браузър:** `{"door": "...", "motor": "...", "esp_connected": true/false}`
- **При грешен token:** Затваря се с код `1008`
- **При свързване:** Сървърът незабавно изпраща текущото пълно съобщение.

---

### 13.3 HTTP статус кодове

| Код | Значение | При кой endpoint |
|-----|----------|-----------------|
| `200 OK` | Успех | Всички успешни заявки |
| `401 Unauthorized` | Невалиден или изтекъл JWT / грешни credentials | `/auth/login`, `/door/*` |
| `409 Conflict` | Моторът вече е в движение | `/door/trigger` |
| `503 Service Unavailable` | ESP32 не е свързан | `/door/trigger` |
| `1008` (WS) | Policy Violation — невалиден token | `/ws/esp`, `/ws/ui` |
