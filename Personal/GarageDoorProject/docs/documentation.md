# Документация на проекта „Garage Door“

## 1. Общ преглед

Проектът представлява система за дистанционно управление на гаражна врата, базирана изцяло на **Bluetooth Low Energy (BLE)**. Състои се от две части:

- **Вграден софтуер** за микроконтролер **ESP32**, който управлява мотора през H-мост и излага BLE GATT услуга.
- **Уеб страница** (`frontend/ble.html`), която се отваря в браузър с поддръжка на Web Bluetooth и комуникира директно с устройството.

Комуникацията е от тип „peer-to-peer“ — няма сървър, няма WiFi връзка, няма облачна услуга. Браузърът говори директно с ESP32 през BLE.

## 2. Архитектура

```
┌─────────────────────┐      BLE GATT      ┌──────────────────────┐     PWM + GPIO    ┌────────┐
│  Браузър + ble.html │  ◀──────────────▶  │  ESP32 (GarageDoor)  │ ────────────────▶ │ H-мост │ ─▶ Мотор
└─────────────────────┘   Write/Notify     └──────────────────────┘                   └────────┘
                                                    │
                                                    ▼
                                           NVS (persistent state)
```

Състоянието на вратата (`open` / `closed` / `unknown`) се пази в енергонезависимата памет (NVS) на ESP32, така че след рестарт устройството помни последната позиция.

## 4. Хардуер

### 4.1. Необходими компоненти

- Платка **ESP32 Dev Module** (или съвместима).
- Драйвер за DC мотор с H-мост (напр. L298N, TB6612FNG, DRV8871).
- Отделно захранване за мотора (препоръчително 5 V / 2 A или повече). **Не захранвайте мотора от USB линията на ESP32** — пусковият ток може да предизвика срив на платката.
- Бутон за фабричен ресет (или се ползва вграденият BOOT бутон).
- Светодиод за индикация (обикновено вграденият на GPIO 2).

### 4.2. Разпределение на пиновете

| Сигнал | GPIO | Предназначение |
|---|---:|---|
| `MOTOR_PIN1` | 27 | Вход 1 на H-моста (посока) |
| `MOTOR_PIN2` | 26 | Вход 2 на H-моста (посока) |
| `ENABLE_PIN` | 14 | PWM enable на H-моста |
| `RESET_BTN_PIN` | 0 | Бутон за фабричен ресет (BOOT бутонът) |
| `LED_PIN` | 2 | Светодиод за статус |

### 4.3. PWM параметри

- Честота: **30 kHz**
- Разделителна способност: **8 бита** (0–255)
- Канал: **0**
- Работен цикъл (`DUTY_CYCLE`): **200** (≈ 78 %)

Моторът се задвижва винаги **за 2 секунди** на едно задействане. Посоката се избира автоматично според текущото състояние на вратата.

## 5. Фърмуер (`GarageDoor.ino`)

### 5.1. Зависимости

- **Arduino Core за ESP32 v3.x** (необходим за новия API `ledcAttachChannel`).
- **NimBLE-Arduino v2.2.x** от h2zero (инсталира се през Library Manager).

Други външни библиотеки не са нужни.

### 5.2. BLE услуга и характеристики

Устройството излага една GATT услуга с четири характеристики:

| Характеристика | UUID | Свойства | Предназначение |
|---|---|---|---|
| Service | `6a4e3200-667b-11ee-b962-0242ac120002` | — | Главна услуга |
| Auth | `6a4e3201-667b-11ee-b962-0242ac120002` | WRITE | Клиентът записва паролата |
| Command | `6a4e3202-667b-11ee-b962-0242ac120002` | WRITE | Команди към устройството |
| State | `6a4e3203-667b-11ee-b962-0242ac120002` | READ + NOTIFY | Текущо състояние (JSON) |
| Info | `6a4e3204-667b-11ee-b962-0242ac120002` | READ | Мета информация (JSON) |

### 5.3. Рекламиране (advertising)

- Име на устройството: `GarageDoor-XXXX`, където `XXXX` са последните 16 бита от MAC адреса.
- Интервали: `20 ms` — `40 ms`.
- Флагове: `0x06` (LE General Discoverable + BR/EDR Not Supported).
- MTU: `185` байта.
- Service UUID се помества в основния пакет, а името — в scan response (в противен случай 31-байтовият лимит на PDU се надхвърля).

### 5.4. Модел на сигурност

1. **Link layer: „Just Works“** — без bonding, без MITM защита, без Secure Connections. Това е умишлено решение, защото Web Bluetooth не показва диалог за pairing, ако устройството няма по-високо ниво на сигурност.
2. **Парола на приложно ниво** — `GARAGE_PASSWORD`, дефинирана в изходния код. Клиентът я записва в Auth характеристиката веднага след свързване.
3. **Флаг за автентикация на всяка връзка поотделно** — всеки клиент получава свое състояние.
4. **Ограничение при неуспешни опити** — `MAX_AUTH_ATTEMPTS = 5`. След пет грешни пароли сесията се прекъсва.
5. Сравнението на паролата е **constant-time**, за да не изтича информация през времето за отговор.

### 5.5. Команди

Командите се изпращат като UTF-8 низове към Command характеристиката. Неавтентикирани заявки се игнорират мълчаливо.

| Команда | Ефект |
|---|---|
| `trigger` | Пуска мотора за 2 секунди и обновява състоянието на вратата |
| `clear_bonds` | Изтрива NVS данните и BLE bonds, после рестартира устройството |

За да не блокира BLE стека, изпълнението и на двете команди се отлага до следващото завъртане на `loop()`.

### 5.6. Формат на състоянието

State характеристиката връща JSON, уникален за всяка връзка (флагът `authed` отразява конкретния клиент):

```json
{
  "door":   "open" | "closed" | "unknown",
  "motor":  "idle" | "moving",
  "authed": true | false
}
```

Info характеристиката връща статична информация:

```json
{
  "firmware": "1.0.0-ble",
  "name":     "GarageDoor-XXXX"
}
```

### 5.7. Логика на мотора

- От състояние `closed` → посока `FORWARD` → ново състояние `open`.
- От състояние `open` → посока `BACKWARD` → ново състояние `closed`.
- От състояние `unknown` → посока `FORWARD` → остава `unknown`.
- Докато `motorBusy` е `true`, всички нови `trigger` команди се отхвърлят.

### 5.8. Съхранение на състоянието

Използва се NVS през библиотеката `Preferences`:

- Namespace: `garage`
- Ключ: `door`
- Стойности: `open`, `closed`, `unknown` (при първоначално четене `unknown` се нормализира до `closed`)

### 5.9. Индикация през светодиода

- **Свети постоянно:** поне един клиент е свързан.
- **Изгасен:** няма свързани клиенти.
- **4 бързи мигания (по 70 ms):** успешна автентикация.

### 5.10. Фабричен ресет

При натиснат BOOT бутон (GPIO 0) по време на или веднага след включване, задържан **≥ 3 секунди**:

1. Изчиства се namespace-ът `garage` в NVS.
2. Извиква се `NimBLEDevice::deleteAllBonds()`.
3. Платката рестартира.

## 6. Уеб клиент (`frontend/ble.html`)

### 6.1. Технология

- Само един HTML файл — няма build система, няма external assets.
- Чист JavaScript, без framework-и.
- Използва `navigator.bluetooth` (Web Bluetooth API).

### 6.2. Конфигурационни константи

В началото на `<script>` блока се намират:

```javascript
const SERVICE_UUID = '6a4e3200-667b-11ee-b962-0242ac120002';
const AUTH_UUID    = '6a4e3201-667b-11ee-b962-0242ac120002';
const CMD_UUID     = '6a4e3202-667b-11ee-b962-0242ac120002';
const STATE_UUID   = '6a4e3203-667b-11ee-b962-0242ac120002';
const DEFAULT_PASSWORD = 'garage-demo-2026';
```

Тези стойности **трябва да съвпадат точно** с тези във фърмуера.

Паролата може временно да се подмени през URL параметъра `?pw=...`, което е удобно за тестване на грешни пароли без редеплой.

### 6.3. Потребителски интерфейс

- **Индикатор за връзка** (цветна точка и етикет):
  - червено — не е свързан;
  - оранжево — свързан, но не е автентикиран;
  - зелено — свързан и автентикиран.
- **Три статус бейджа**: Door, Motor, Auth — с цветове, съответстващи на състоянието.
- **Голям кръгъл бутон** в средата, чийто надпис се променя според състоянието (`Connect first`, `Authenticating…`, `Moving…`, `Open`, `Close`).
- **Три вторични бутона**: Connect, Disconnect, Forget device.
- **Activity log** — монохромен панел в долната част с времеви маркери на всички операции и събития; много полезен при дебъгване.

Ако браузърът не поддържа Web Bluetooth, целият UI се заменя със съобщение и списък на съвместимите платформи.

### 6.4. Работен поток

1. Потребителят натиска **Connect**.
2. Извиква се `navigator.bluetooth.requestDevice()` с филтър `namePrefix: 'GarageDoor-'`.
3. След избор на устройство се извършва `device.gatt.connect()`.
4. Клиентът взема услугата и трите характеристики (Auth, Command, State).
5. Абонира се за нотификации на State характеристиката.
6. Записва паролата в Auth характеристиката.
7. Получава обратно State нотификация с `authed: true` или `authed: false`.

След успешна автентикация големият бутон става активен и изпраща `trigger` при натискане.

## 7. Компилация и флашване

### 7.1. С arduino-cli

```bash
arduino-cli compile --fqbn esp32:esp32:esp32 embedded/GarageDoor
arduino-cli upload  --fqbn esp32:esp32:esp32 -p /dev/cu.usbserial-XXXX embedded/GarageDoor
```

Заменете порта с този, който се показва при свързването на платката.

### 7.2. С Arduino IDE

1. Отворете `embedded/GarageDoor/GarageDoor.ino`.
2. Изберете борда **ESP32 Dev Module**.
3. Изберете правилния сериен порт.
4. Натиснете **Upload**.

### 7.3. Сериен монитор

Скорост: **115200 baud**. При нормален старт трябва да видите нещо от рода на:

```
Door state loaded: closed
Starting BLE as 'GarageDoor-ABCD'
BLE advertising started. Waiting for a client...
```

## 8. Хостване на уеб клиента

Web Bluetooth работи само в **защитен контекст** (HTTPS или `localhost`). Няколко варианта:

- **Netlify Drop** — най-бързият начин, просто се издърпва файлът към [app.netlify.com/drop](https://app.netlify.com/drop).
- **GitHub Pages** — копирайте `ble.html` в `docs/` и активирайте Pages.
- **Cloudflare Pages / Vercel** — за автоматичен деплой от хранилището.
- **Локално**: от папката `frontend/` изпълнете `python3 -m http.server 8000` и отворете `http://localhost:8000/ble.html`.

## 9. Демонстрация стъпка по стъпка

1. Включете ESP32 — в серийния монитор трябва да се появи „BLE advertising started“.
2. Отворете `ble.html` в съвместим браузър.
3. Натиснете **Connect** — системният диалог показва `GarageDoor-XXXX`.
4. Изберете устройството. Страницата автоматично записва паролата и бейджът **Auth** сменя състоянието си на **OK**.
5. Натиснете големия бутон — моторът работи 2 секунди, а бейджът **Door** се превключва.
6. За повторна демонстрация на pairing може:
   - да натиснете **Forget device** в страницата (изпраща `clear_bonds`);
   - или да задържите BOOT бутона за 3 секунди;
   - допълнително изтрийте устройството и от настройките на ОС (macOS / Windows / Android / iOS имат различни менюта).

## 10. Настройка и персонализация

### 10.1. Смяна на паролата

1. Променете `GARAGE_PASSWORD` в `GarageDoor.ino` и флашнете отново.
2. Променете `DEFAULT_PASSWORD` в `ble.html` и редеплойнете страницата.

Двете стойности задължително трябва да съвпадат.

### 10.2. Уникални UUID-та за няколко устройства

Ако имате повече от едно устройство във fleet, генерирайте нов набор UUID-та:

```bash
uuidgen                              # macOS / Linux
python -c "import uuid; print(uuid.uuid4())"
```

След това обновете петте дефиниции (`SERVICE_UUID`, `AUTH_UUID`, `CMD_UUID`, `STATE_UUID`, `INFO_UUID`) и в `GarageDoor.ino`, и в `ble.html`.

## 11. Бележки по сигурността

Настоящата реализация е подходяща за **демонстрация и прототипиране**, а не за продукционно инсталиране пред реален гараж. Основните рискове са:

- **„Just Works“ link-layer**: всеки, който е в обхват и знае паролата, може да се свърже.
- **Паролата се предава в чист вид** по нешифрован BLE канал — може да бъде прихваната от BLE sniffer.
- Защитата срещу brute-force е проста — 5 грешни опита и дисконект на конкретната връзка, но нищо не пречи клиентът да се свърже отново.

За продукция би трябвало да се добавят: BLE bonding с LE Secure Connections, подписани команди с nonce и challenge-response автентикация.

## 12. Отстраняване на проблеми

| Симптом | Вероятна причина | Решение |
|---|---|---|
| В системния диалог не се появява устройство | ESP32 не рекламира, извън обхват или вече свързан с друг клиент | Рестартирайте ESP32, проверете серийния лог |
| Свързването работи, но Auth остава на „…“ | Различни пароли между фърмуер и страница | Сверете `GARAGE_PASSWORD` и `DEFAULT_PASSWORD` |
| macOS не вижда устройството стабилно | Проблем с Bluetooth стека на macOS | Изпълнете `sudo pkill bluetoothd` |
| Прекъсващи връзки | 2.4 GHz смущения между BLE и WiFi | Изключете WiFi на лаптопа за тестовете |
| Моторът не се върти | Грешно окабеляване или липсва захранване | Проверете H-моста, захранването и PWM сигнала |
| `navigator.bluetooth is undefined` | Браузърът не поддържа Web Bluetooth | Ползвайте Chrome/Edge или Bluefy (iOS) |
